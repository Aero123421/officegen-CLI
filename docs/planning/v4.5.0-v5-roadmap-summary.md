# Officegen-CLI v4.5.0 to v5 Roadmap Summary

Purpose: summarize what v4.5.0 must prove now and what should intentionally move to v5. This document complements `docs/planning/v4.5.0-acceptance-matrix.md`.

## v4.5.0 Release Thesis

v4.5.0 should be a trustworthy native Rust transition release: Node-free for normal CLI execution, conservative about Office/PDF fidelity, explicit about unsupported surfaces, and useful for agents through compact capabilities, stable envelopes, structured errors, schemas, and an inspect/edit/view/verify loop.

The release is successful when it proves the following:

- Command registry, help, capabilities, and dispatch agree.
- MCP remains excluded from runtime, help, capabilities, release smoke, and v5 scope.
- Capabilities are compact enough for agents while still disclosing limitations.
- JSON envelopes, errors, and schema discovery remain stable.
- Path and ZIP safety fail closed.
- Inspect, render, edit, view, verify, diff, and manifest smoke produce durable evidence.
- Release gates pass without reintroducing Node.js as a runtime requirement for native assets.

## v4.5.0 Must-Haves

| Theme | Required acceptance IDs | Release intent |
| --- | --- | --- |
| Command registry | V45-CMD-001 through V45-CMD-004 | Make command discovery reliable for agents and humans. |
| MCP exclusion | V45-MCP-001 through V45-MCP-003 | Keep MCP out of the native CLI scope entirely. |
| Compact capabilities | V45-CAP-001 through V45-CAP-003 | Keep the capability contract small, honest, and progressive. |
| Envelope/error/schema | V45-ENV-001 through V45-ENV-004 | Preserve machine-readable success, failure, validation, and recovery behavior. |
| Path/ZIP safety | V45-SAFE-001 through V45-SAFE-004 | Block unsafe file and package handling before semantic work begins. |
| Core smoke loop | V45-SMOKE-001 through V45-SMOKE-007 | Demonstrate the agent-facing document loop and evidence trail. |
| Release DoD | V45-REL-001 through V45-REL-005 | Ensure version sync, native assets, Rust checks, Node legacy checks, and review artifacts are complete. |

## Non-Goals for v4.5.0

These should not block v4.5.0 if documented and represented in structured output:

- General PDF content rewriting, physical redaction, and full typographic PDF fidelity.
- Cross-format native Office conversion in the portable default runtime.
- MCP server support in any built-in profile.
- Full v3.1.x parity for every template, design, layout, asset, chart, and workflow mutation.
- Network, cloud API, Chromium, Java, Python, Office, or LibreOffice as a default runtime dependency.

## v5 Entry Criteria

Begin v5 implementation only after v4.5.0 has either passed or explicitly waived the required `V45-*` acceptance IDs with reviewer notes. Each v5 roadmap item should add its own acceptance IDs before source implementation begins, so new surfaces cannot ship as undocumented command behavior.
